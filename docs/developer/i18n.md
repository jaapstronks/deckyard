# Internationalization (i18n)

This guide covers the UI internationalization system for the presentation system.

## Directory Structure

```
client/i18n/
├── manifest.json           # List of available locales (12)
├── shared.json             # Dead at runtime — see "shared.json" below
├── template.pot.json       # Extraction template
├── en/                     # English (reference language)
│   ├── common.json         # Common UI strings
│   ├── auth.json           # Login, forgot/reset password
│   ├── editor.json         # Editor interface
│   ├── list.json           # Presentation list, libraries
│   ├── share.json          # Sharing, viewer, moderation
│   ├── settings.json       # Settings, admin panel
│   ├── presenter.json      # Presenter mode
│   ├── slide-types.json    # Slide type labels/options
│   └── follow.json         # Follow-view chrome (en + nl only; separate loader)
├── nl/                     # Dutch
│   └── (same structure)
├── de/, fr/, es/, it/, pl/, fi/, pt/, da/, sv/, no/
│   └── (same structure, minus follow.json)
```

The app loads the modular `<locale>/<component>.json` files directly
(`client/lib/ui-i18n.js`, `I18N_COMPONENTS`). There is no merged
`index.json`/`en.json` build artifact anymore — if you see one, it's stale
and gitignored.

## Module Organization

| Module | Categories | Purpose |
|--------|------------|---------|
| `shared.json` | — | Colour names, symbols, brand names. **Not loaded at runtime** — see below |
| `common.json` | common, app, follow | Generic UI actions |
| `auth.json` | login, forgotPassword, resetPassword | Authentication flows |
| `editor.json` | editor | Editor interface |
| `list.json` | list, slideLibrary, imageLibrary | Lists and libraries |
| `share.json` | share, shareViewer, moderate | Sharing features |
| `settings.json` | settings, admin | User/admin settings |
| `presenter.json` | presenter, notesJoin | Presenter mode |
| `slide-types.json` | slideType | Slide type labels |
| `follow.json` | follow | Follow-view chrome — **not** loaded via `t()`; a scoped loader (`client/views/follow/i18n.js`) resolves it against the *deck* language rather than the UI locale |

## Key Conventions

Use dot-notation keys with category prefix:

```javascript
t('common.save', 'Save')
t('editor.share.title', 'Share presentation')
t('slideType.textSlide.label', 'Text Slide')
```

## Using Translations

Import and call `t()`:

```javascript
import { t } from '/client/lib/ui-i18n.js';

// Simple translation
const label = t('common.save', 'Save');

// With interpolation
const msg = t('list.count', '{count} presentations', { count: 5 });
```

The second argument is the English fallback, used when:
- Translation is missing for current locale
- File fails to load

### shared.json

`client/i18n/shared.json` holds ~200 language-agnostic strings, but **nothing
at runtime reads it** — `ui-i18n.js` fetches only the per-locale module files.
Only the build scripts touch it, and `i18n-validate.js` used to count its keys
as "covered", which hid real gaps. Treat it as legacy: don't add keys to it,
and put a string in the locale files even when it looks language-agnostic.

## Adding New Strings

Write the call site first — the English fallback is the source of truth:

```javascript
t('editor.foo.bar', 'Some English text')
```

Then materialize the keys. English is mechanical (the fallback *is* the
English string); Dutch needs real translation:

```bash
node scripts/i18n-fill.js en              # write missing EN keys from fallbacks
node scripts/i18n-fill.js --report nl     # dump missing NL keys as JSON
node scripts/i18n-fill.js --apply nl out.json   # merge translations back
```

`npm test` fails if a key used in code is missing from `nl/` or `en/`, so this
is not optional — see `tests/i18n-coverage.test.js`.

**Never build a translatable string by concatenation.** A key must contain the
whole sentence, with `{var}` placeholders for the moving parts, because word
order differs per language:

```javascript
t('x.days', '{n} days', { n: 30 })            // ✅
'30 ' + t('x.days', 'days')                   // ❌ untranslatable
t('x.del', `Delete "${name}"?`)               // ❌ only the fallback interpolates
```

## Scripts

| Script | Purpose |
|--------|---------|
| `npm run i18n:audit` | Find hardcoded copy that bypasses `t()`, and orphan keys |
| `node scripts/i18n-fill.js` | Materialize EN from fallbacks; report/apply other locales |
| `node scripts/i18n-validate.js` | Check for JSON errors, missing keys, empty values |
| `node scripts/i18n-extract.js` | Extract keys from source code |
| `node scripts/i18n-sync.js` | Fill missing keys with English placeholders |
| `node scripts/i18n-modularize.js` | One-time script to split flat files |

`scripts/i18n-keys.js` is the shared extractor both the fill script and the
coverage test use, so they always agree on what "a key the code uses" means.

### What the two gates each catch

`tests/i18n-coverage.test.js` guards the **keys**: everything the code asks for
must exist in `nl/` and `en/`. It is blind to copy that never asks — a label
written as a literal inside `h(...)` has no key to be missing, so it renders
English in all 12 locales and no test notices.

`tests/i18n-audit.test.js` closes that hole. `npm run i18n:audit` scans for
user-facing literals (`text:`, `title:`, `label:`, `placeholder:`, `alt:`,
`aria-label:`, `hint:`, `toast(…)`) and fails on anything not listed in
`scripts/i18n-audit-allowlist.json`. That file is a **burndown**, in the same
spirit as `eslint-suppressions.json`: each entry needs a reason, stale entries
fail the build, and the list is meant to shrink.

Legitimate reasons to allowlist rather than translate: brand and product names
(Unsplash, Matomo), language names shown in their own language, language tags
(`NL`), syntax examples (`Raleway:400,700`), and typographic specimens (`Aa`).

Two patterns are *not* findings, and the scanner already skips both:

```javascript
// 1. literal as the t() fallback
h('div', { text: t('common.save', 'Save') })

// 2. config table built at import time, before the dictionary loads —
//    the sibling *Key prop is what gets resolved at render time
const SOURCES = [{ key: 'upload', labelKey: 'fonts.source.upload', label: 'Upload Custom Font' }];
h('span', { text: t(src.labelKey, src.label) });
```

### Orphan keys

`npm run i18n:audit --orphans` lists keys present in `en/` but referenced
nowhere. There are **~268** of them, left behind by call sites that moved or
died. They are harmless at runtime but inflate all 12 locales and make "how
translated are we?" unanswerable, so they are reported but **not** gated —
pruning them is tracked as its own task. The check is deliberately
conservative: a key referenced by any dotted string literal anywhere in
`client/`, `shared/`, `server/`, `custom/` or `themes/`, or matching a
runtime-built family in `DYNAMIC_KEY_PATTERNS`, is never called an orphan.

(`scripts/i18n-merge.js` still exists but builds an artifact the runtime no
longer reads; don't use it. `i18n-sync.js` copies English text into other
locales, which makes them *look* translated — prefer `i18n-fill.js --report`.)

## Dutch style

Dutch here borrows English freely; don't over-translate product vocabulary.

| Use | Not |
|-----|-----|
| slide, slides | dia, dia's |
| slide-type, slide-bibliotheek | diatype, diabibliotheek |
| reactie (a comment) | opmerking |
| antwoord (a reply to a comment) | reactie |
| workspace, lead, analytics | werkruimte, … |

Address the user informally (`je` / `jouw`, never `u`), use infinitives for
action labels (Opslaan, Annuleren, Verwijderen), and sentence case for
headings — Dutch does not capitalize every word.

## File Size

There is **no line limit on translation files**. The ~600-line rule of thumb
applies to hand-written files that carry logic — mostly JavaScript modules —
where length is a proxy for "too much going on to hold in your head". A locale
file is a flat, generated key/value map: it has no branching to follow, it is
never read top to bottom, and it grows by one line per key whether you like it
or not. Splitting it buys nothing and costs an entry in `I18N_COMPONENTS`
(`client/lib/ui-i18n.js`) plus one more HTTP fetch per page load.

`scripts/i18n-validate.js` used to fail the build over this and flagged every
locale on `slide-types.json`; it no longer checks line counts.

## Supported Languages

`en` and `nl` are the first-class locales: both define **every** key the code
uses, and `npm test` keeps them that way. The rest are partial — roughly 900
keys behind — and fall back to the English string in each `t()` call, so they
degrade to English rather than breaking.

## Shared Strings

The `shared.json` file contains ~200 keys that are identical across all languages:
- Color names: "lime", "mist", "gray"
- Numbers: "1", "2", "3", etc.
- Symbols: "—" (em dash)
- Brand names: "Notion", "Markdown"

These are loaded once and merged with language-specific translations.

## RTL (Right-to-Left) Text Support

Slide content supports RTL languages (Arabic, Hebrew, Persian, Urdu) automatically. Users can type RTL text in any slide field and it will render correctly.

### How it works

All text elements in slide HTML use `dir="auto"`, which lets the browser detect text direction from the first strong directional character. This means:

- **Arabic/Hebrew text** → renders right-to-left
- **English/Dutch text** → renders left-to-right
- **Mixed content** → each element determines its own direction

No manual toggle is needed. Users just type in their preferred language.

### Technical implementation

1. **Slide type templates** (`shared/slide-types/types/*.js`): All text elements (`<h1>`, `<h2>`, `<p>`, `<div>` with text) include `dir="auto"`

2. **Markdown renderer** (`shared/markdown.js`): Generated elements (`<p>`, `<ul>`, `<ol>`, `<li>`, `<th>`, `<td>`) include `dir="auto"`. Code blocks keep `dir="ltr"` to preserve syntax direction.

3. **CSS** (`client/styles/slides/`): Uses `text-align: start` instead of `text-align: left` for direction-aware alignment

4. **HTML export** (`server/export/html.js`): Sets `dir` attribute on `<html>` element based on document language via `getDocDir()` from `server/utils/doc-lang.js`

### Supported RTL language codes

| Code | Language |
|------|----------|
| ar | Arabic |
| he | Hebrew |
| fa | Persian (Farsi) |
| ur | Urdu |

### Adding a new slide type

When creating a new slide type, ensure all user-facing text elements include `dir="auto"`:

```javascript
renderHtml: (content) => `
  <div class="slide slide-example">
    <h2 class="heading" dir="auto">${esc(content?.title)}</h2>
    <p class="body" dir="auto">${esc(content?.body)}</p>
  </div>
`
```
